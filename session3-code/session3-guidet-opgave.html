💻 Dag 1 · Session 3 · Claude Code

Byg en rigtig app — og lad Claude tjekke sit eget arbejde

Nu skifter vi værktøj. Vi går fra chat-vinduet og dokumentmappen over i terminalen, hvor Claude Code bygger en faktisk app: React-frontend, FastAPI-backend, SQLite. Samme NordPension-opgave, men nu som software der kører på din maskine — og hvor Claude selv kan inspicere det den lige har lavet.

Fase 1-4 gennemgår vi sammen på storskærm — du følger med i din egen terminal. Fra fase 5 arbejder du selv.

Sessionens fem faser

  1. Start et nyt Claude Code-projekt — terminal, stack, scaffold
  2. Vælg din dataadgangs-strategi — MCP eller direkte Supabase
  3. Byg "Min medlemsportefølje"-appen — list + detail
  4. Installer frontend-design skill — og lad Claude redesigne
  5. Playwright self-validation — den agentiske loop (solo)
1

Start et nyt Claude Code-projekt

Vi forlader browseren. Claude Code lever i terminalen og kan læse, skrive og køre kode direkte i en mappe på din maskine.

1.1 — Åbn terminal og start Claude
Åbn en terminal og lav en ny, tom mappe — fx mkdir nordpension-portal && cd nordpension-portal.
Start Claude Code med kommandoen claude. Du får et prompt-felt i terminalen.

Claude Code arbejder altid relativt til den mappe du startede den i. Den kan ikke se filer udenfor.

1.2 — Bed Claude scaffolde projektet

Første prompt definerer hele stakken. Vi vælger noget bevidst småt og pragmatisk:

Bootstrap et nyt projekt i denne mappe med følgende stack: - Frontend: React + Vite + TypeScript (brug yarn som package manager) - Backend: FastAPI (Python, brug uv som package manager) - Database: SQLite lokalt Lav en simpel "hello world"-version end-to-end så jeg kan se at det hele kører. Forklar undervejs hvad du laver, og bed mig om lov før du installerer pakker.
Claude opretter en mappestruktur (typisk frontend/ og backend/), skriver scaffold-filer, kører yarn og uv-kommandoer og beder om din godkendelse undervejs.
Den agentiske forskel: I chat fik du forslag. Her gør Claude tingene. Den beder om lov, læser filer, kører kommandoer — og du holder hånden på rattet ved at sige ja eller nej.
1.3 — Få det op at køre
Start både backend og frontend så jeg kan åbne appen i browseren. Fortæl mig hvilken URL den kører på.
Claude starter uvicorn (FastAPI) og yarn dev (Vite) i baggrunden og giver dig en localhost-URL. Du åbner den og ser "hello world".
Hvis noget fejler, så sig "Det virker ikke — fejlbeskeden er: …" og lim outputtet ind. Claude læser fejlen, retter og prøver igen.
2

Vælg din dataadgangs-strategi

Før vi bygger noget rigtigt, skal vi træffe ét eksplicit valg: hvor kommer data fra? I session 1 var det givet — MCP-connectoren. Her har du to muligheder, og koden flytter ansvaret over på dig.

2.1 — De to veje

Vej A — via NordPension MCP-connectoren. Samme endpoint som i session 1: https://nordpension-mcp.jeprasher.workers.dev/mcp. PII er filtreret væk på serversiden. Governance kommer gratis med.

Vej B — direkte mod Supabase REST API. URL: https://pruvwgulvquauncykgxy.supabase.co/rest/v1/ med en anon key. Du får alt — inklusive navne, CPR, email, adresser. Til gengæld er det dit ansvar hvad du gør med det.

Pointen: Når du bygger en app, kan du selv vælge hvor stramt governance-laget skal sidde. MCP'en filtrerer for dig. Den direkte vej gør ikke. Det er et reelt design-valg, ikke en teknisk detalje.
2.2 — Lad Claude argumentere for begge sider
Jeg skal vælge mellem to dataadgangs-strategier til "Min medlemsportefølje"-app: A) NordPension MCP-connector (PII-filtreret, governance indbygget) Endpoint: https://nordpension-mcp.jeprasher.workers.dev/mcp B) Supabase REST direkte (anon key, fuld data inkl. PII) Endpoint: https://pruvwgulvquauncykgxy.supabase.co/rest/v1/ Argumentér seriøst for begge. Hvad vinder jeg, hvad mister jeg, hvad bliver mit ansvar i hver?
Claude giver dig en afvejning: MCP = sikkert default men begrænset; direkte = fuld fleksibilitet men du skal selv tænke på PII, logging, samtykke, lethal trifecta osv.
2.3 — Træf valget og dokumentér det
Jeg vælger [A / B]. Opret en README.md i projektets rod der dokumenterer valget, ræsonnementet og hvilke konsekvenser det har for hvordan vi håndterer data fremover.
Du har nu en README med et eksplicit governance-spor. Når en kollega — eller revisor — kigger om tre måneder, kan de se hvorfor.
Hvis du vælger MCP-vejen, så bed Claude om at tilføje connectoren til .claude/settings.json i projektet, så den er tilgængelig som værktøj i Claude Code.
3

Byg "Min medlemsportefølje"-appen

Nu det rigtige. Appen er bevidst lille: en liste over medlemmer og en detail-side per medlem med deres sager. Det er nok til at mærke hele loopet.

3.1 — Beskriv funktionaliteten
Byg "Min medlemsportefølje" end-to-end: Backend (FastAPI): - GET /api/members → liste af medlemmer (id, age, profession_group, region, annual_salary) - GET /api/members/{id} → ét medlem + dets aktive sager - GET /api/cases?member_id={id} → sager for et medlem Data henter du via [MCP-connectoren / Supabase REST] ud fra mit valg i README. Brug gerne SQLite som lokal cache hvis det giver mening, ellers proxy direkte igennem. Frontend (React + Vite): - Forsiden: en tabel med medlemmer, klikbar - Detail-siden: medlemsoplysninger øverst, sager nedenunder Gør det enkelt. Ingen auth, ingen state-management bibliotek. Bare list + detail. Når du er færdig, start serverne og verificér at appen virker ved at hente /api/members og kigge på output.
Claude skriver routes, en frontend, og kører selv en curl mod sit eget endpoint for at tjekke at data kommer ud. Hvis noget fejler, fejlsøger den selv før den giver op.
3.2 — Åbn den i browseren
Åbn appen på localhost. Klik dig rundt. Kig på en medlemsdetalje.
Det virker — men det ser typisk meget "default" ud. Hvid baggrund, sort tekst, standard browser-fonts. Funktionelt, men kedeligt.
Det er pointen. Vi vil mærke kontrasten i næste fase, når Skills træder ind.
4

Installer frontend-design skill og regenerér UI'en

Samme trick som theme-factory i session 1 — bare i Code-kontekst. Skills lærer Claude konventioner og hæver kvaliteten gratis.

4.1 — Installer skillen
Installer skillen "frontend-design" fra https://www.skills.sh/anthropics/skills/frontend-design i dette projekt. Bekræft at den er tilgængelig.
Claude henter skillen, lægger den i .claude/skills/ og bekræfter at den nu er aktiv.

Skillen er Anthropics egen samling af principper for "ikke at lave generisk AI-look" — typografi, spacing, hierarki, farvevalg.

4.2 — Tag et "før"-screenshot
Inden du redesigner: tag et screenshot af appen som den ser ud nu. Gem det som before.png i projektet.
Du kan også bede Claude om at gøre det med Playwright — så har du allerede første tråd til fase 5.
4.3 — Redesign med skillen aktiv
Redesign frontend'en til "Min medlemsportefølje" med frontend-design-skillen aktiv. Behold funktionaliteten, men løft kvaliteten af typografi, layout og hierarki. Brug en seriøs, dæmpet visuel tone — det er et internt værktøj, ikke en forbrugerapp.
Claude opdaterer CSS/komponenter, kører dev-serveren igen, og du ser en mærkbart mere moden UI: konsistent spacing, ordentlig type-skala, gennemtænkte farver.
Læringen, igen: Skills er det, der gør hver bruger god uden at hver bruger selv skal være designer. Det er den samme idé som i chat-sessionen — bare i kode.
5

Solo: Playwright self-validation

Her bliver det rigtigt agentisk. Du beder Claude bygge en test der åbner appen, tager et screenshot, og kigger på sit eget skærmbillede for at vurdere om det ligner det den troede den havde bygget.

Solo-opgave

Få Claude til at inspicere sit eget arbejde

Vi har bygget en app. Nu skal vi lukke loopet: koden, det visuelle output, og en vurdering af om de matcher. Det er den agentiske loop i miniature — og den du skal øve dig i at have tillid til.

5.1 — Sæt Playwright op
Installer Playwright i projektet. Skriv en test der: 1. Starter dev-serveren (eller forventer at den allerede kører) 2. Åbner appen på localhost 3. Tager et screenshot af forsiden og en detail-side 4. Gemmer screenshots i mappen ./screenshots/ Kør testen og bekræft at billederne faktisk findes.
Du har en kørende test, og to PNG-filer i ./screenshots/.
5.2 — Lad Claude læse sit eget output
Læs nu screenshots/forside.png og screenshots/detail.png. Beskriv hvad du ser. Stemmer det med det jeg bad dig bygge i fase 3? Er der noget der ser brudt, forkert eller "AI-genereret klodset" ud?
Claude åbner billedfilerne, beskriver konkret hvad der er på skærmen, sammenligner med specifikationen, og peger på ting den selv synes ser skævt ud.
Det her er kerneoplevelsen: Claude lukker selv loopet mellem "jeg skrev kode" og "jeg ved hvordan det ser ud". Det er meget tæt på hvordan en juniorudvikler arbejder — bare hurtigere og uden trætte øjne.
5.3 — Stretch: Tilføj en ekstern datakilde

Hvis du har tid og lyst — udvid appen med data fra en ekstern API. Forslag:

  • Danmarks Statistik StatBank API — fx pensionsformue eller arbejdsmarkedstal som benchmark
  • Nationalbankens statistik-API — IP-statistik eller renter
Find en relevant ekstern API (fx Danmarks Statistik) og tilføj et benchmark-tal til medlems-detailsiden, så jeg kan sammenligne et medlems opsparing mod et landsgennemsnit. Hvis API'et kræver en nøgle, så læg den i en .env-fil og sørg for at den ikke committes.
Når Claude rører ved .env, hemmeligheder eller eksterne kald, så sig stop og tænk en gang: hvad kan en kombination af "læser data + ekstern netværksadgang + utrusted input" føre til? Den tankerække skal vi tale om bagefter — du skal bare have plantet kimen nu.
5.4 — Hvor langt nåede du?

Du vælger selv ambitionsniveau:

  • Minimum: Playwright + self-validation kører
  • Mellem: Også før/efter-sammenligning af UI'en
  • Fuld: Ekstern API med nøgle i .env + benchmark på detail-siden

Vi samles op bagefter. Hav gerne en konkret oplevelse med — noget der overraskede dig, noget der gik galt, noget Claude foreslog du ikke selv havde tænkt på.
