💻 Dag 1 · Session 3 · Claude Code

Byg en rigtig app — og lad Claude tjekke sit eget arbejde

Nu skifter vi værktøj. Vi går fra chat-vinduet og dokumentmappen over i Claude Desktop-appens Code-tab, hvor Claude bygger en faktisk app: React-frontend, FastAPI-backend, SQLite. Samme NordPension-opgave, men nu som software der kører på din maskine — og hvor Claude selv kan inspicere det den lige har lavet via den indbyggede preview.

Fase 1-4 gennemgår vi sammen på storskærm — du følger med i din egen Code-session. Fra fase 5 arbejder du selv.

Sessionens fem faser

  1. Start en ny Code-session — folder, stack, scaffold
  2. Vælg din dataadgangs-strategi — MCP eller direkte Supabase
  3. Byg "Min medlemsportefølje"-appen — list + detail
  4. Installer Playwright — og lad den verificere at appen virker
  5. Solo: frontend-design + før/efter — redesign og dokumentér ændringen
1

Start en ny Code-session

Vi forlader chat-vinduet og dokumentmappen. I Claude Desktop-appens Code-tab er hver session sit eget mini-projekt med en mappe, en chat, en preview og en diff-visning side om side.

1.1 — Åbn Code-tab og start en session
Åbn Claude Desktop-appen og klik på Code-tabben øverst til venstre.
Klik + New session i sidebaren (eller tryk ⌘N).
I prompt-området nederst: hold Local som environment. Klik mappe-chip'en og vælg session3-code-mappen fra workshop-materialet.
Vælg model i dropdown'en til højre (Opus 4.7 er fint). Sæt permission-mode til Ask permissions så du ser hvad Claude laver i starten.

Hver Code-session er isoleret: sin egen mappe, sin egen chat, sin egen diff-historik. Du kan have flere kørende parallelt i sidebaren.

1.2 — Bed Claude scaffolde projektet

Første prompt definerer hele stakken. Vi vælger noget bevidst småt og pragmatisk:

Bootstrap et nyt projekt i denne mappe med følgende stack: - Frontend: React + Vite + TypeScript (brug yarn som package manager) - Backend: FastAPI (Python, brug uv som package manager) - Database: SQLite lokalt Lav en simpel "hello world"-version end-to-end så jeg kan se at det hele kører. Forklar undervejs hvad du laver, og bed mig om lov før du installerer pakker.
Claude opretter en mappestruktur (typisk frontend/ og backend/), skriver scaffold-filer, kører yarn og uv-kommandoer og beder om din godkendelse undervejs.
Den agentiske forskel: I chat fik du forslag. Her gør Claude tingene. Den beder om lov, læser filer, kører kommandoer — og du holder hånden på rattet ved at sige ja eller nej.
1.3 — Få det op at køre i Preview-panelet
Start både backend og frontend. Brug Preview-panelet i Code-tab'en så jeg kan se appen kørende direkte i Claude.
Claude starter uvicorn (FastAPI) og yarn dev (Vite), registrerer dev-serveren i .claude/launch.json, og åbner det indbyggede Preview-panel med appen kørende på localhost.
Brug for terminalen alligevel? Tryk Ctrl+` i Code-tab'en for at åbne den integrerede terminal — den lever i samme arbejdsmappe som Claude.
2

Vælg din dataadgangs-strategi

Før vi bygger noget rigtigt, skal vi træffe ét eksplicit valg: hvor kommer data fra? I session 1 var det givet — MCP-connectoren. Her har du to muligheder, og koden flytter ansvaret over på dig.

2.1 — De to veje

Vej A — via NordPension MCP-connectoren. Samme endpoint som i session 1: https://nordpension-mcp.jeprasher.workers.dev/mcp. PII er filtreret væk på serversiden. Governance kommer gratis med.

Vej B — direkte mod Supabase REST API. URL: https://pruvwgulvquauncykgxy.supabase.co/rest/v1/ med en anon key. Du får alt — inklusive navne, CPR, email, adresser. Til gengæld er det dit ansvar hvad du gør med det.

Pointen: Når du bygger en app, kan du selv vælge hvor stramt governance-laget skal sidde. MCP'en filtrerer for dig. Den direkte vej gør ikke. Det er et reelt design-valg, ikke en teknisk detalje.
2.2 — Lad Claude argumentere for begge sider
Jeg skal vælge mellem to dataadgangs-strategier til "Min medlemsportefølje"-app: A) NordPension MCP-connector (PII-filtreret, governance indbygget) Endpoint: https://nordpension-mcp.jeprasher.workers.dev/mcp B) Supabase REST direkte (anon key, fuld data inkl. PII) Endpoint: https://pruvwgulvquauncykgxy.supabase.co/rest/v1/ Argumentér seriøst for begge. Hvad vinder jeg, hvad mister jeg, hvad bliver mit ansvar i hver?
Claude giver dig en afvejning: MCP = sikkert default men begrænset; direkte = fuld fleksibilitet men du skal selv tænke på PII, logging, samtykke, lethal trifecta osv.
2.3 — Træf valget og dokumentér det
Jeg vælger [A / B]. Opret en README.md i projektets rod der dokumenterer valget, ræsonnementet og hvilke konsekvenser det har for hvordan vi håndterer data fremover.
Du har nu en README med et eksplicit governance-spor. Når en kollega — eller revisor — kigger om tre måneder, kan de se hvorfor.
Vælger du MCP-vejen, så tilføj connectoren via +-knappen ved siden af prompt-feltet → Connectors. Indsæt URL'en https://nordpension-mcp.jeprasher.workers.dev/mcp. Den er så tilgængelig som værktøj resten af sessionen.
3

Byg "Min medlemsportefølje"-appen

Nu det rigtige. Appen er bevidst lille: en liste over medlemmer og en detail-side per medlem med deres sager. Det er nok til at mærke hele loopet.

3.1 — Beskriv funktionaliteten
Byg "Min medlemsportefølje" end-to-end: Backend (FastAPI): - GET /api/members → liste af medlemmer (id, age, profession_group, region, annual_salary) - GET /api/members/{id} → ét medlem + dets aktive sager - GET /api/cases?member_id={id} → sager for et medlem Data henter du via [MCP-connectoren / Supabase REST] ud fra mit valg i README. Brug gerne SQLite som lokal cache hvis det giver mening, ellers proxy direkte igennem. Frontend (React + Vite): - Forsiden: en tabel med medlemmer, klikbar - Detail-siden: medlemsoplysninger øverst, sager nedenunder Gør det enkelt. Ingen auth, ingen state-management bibliotek. Bare list + detail. Når du er færdig, start serverne og verificér at appen virker ved at hente /api/members og kigge på output.
Claude skriver routes, en frontend, og kører selv en curl mod sit eget endpoint for at tjekke at data kommer ud. Hvis noget fejler, fejlsøger den selv før den giver op.
3.2 — Inspicér den i Preview-panelet
Åbn Preview-panelet i Code-tab'en (eller tryk ⌘⇧P). Klik dig rundt i appen — forside, klik på et medlem, se detail-siden.
Det virker — men det ser typisk meget "default" ud. Hvid baggrund, sort tekst, standard browser-fonts. Funktionelt, men kedeligt.
Det er pointen. Vi vil mærke kontrasten i næste fase, når Skills træder ind. Bemærk også at Claude allerede har auto-verificeret sit eget arbejde med screenshots undervejs — auto-verify er slået til som default.
4

Installer Playwright og verificér at appen virker

Code-tab'en har allerede auto-verify — Claude tager screenshots undervejs. Nu løfter vi det til en eksplicit, reproducerbar test du kan køre igen senere, og vi bruger samme test til at fange et "før"-billede af UI'en inden vi rører ved designet i fase 5.

4.1 — Bed Claude installere Playwright og skrive en smoke-test

Du laver ikke noget i terminalen selv. Claude står for hele installationen, konfigurationen og første test-kørsel — du gennemlæser og godkender undervejs.

Installer Playwright i projektet (browser binaries + npm-pakke). Skriv en smoke-test der: 1. Antager at dev-serveren allerede kører (eller starter den hvis nødvendigt) 2. Åbner forsiden af "Min medlemsportefølje" 3. Verificerer at der vises en liste af medlemmer (mindst ét element) 4. Klikker på første medlem og verificerer at detail-siden loader med medlemsdata + sager 5. Tager screenshots af forsiden og detail-siden og gemmer dem som screenshots/before-forside.png og screenshots/before-detail.png Kør testen og bekræft at den passerer og at screenshot-filerne findes.
Claude installerer Playwright, skriver en test-fil, kører npx playwright test, og rapporterer både test-resultatet og at de to PNG'er ligger i ./screenshots/.
4.2 — Lad Claude review sit eget output
Læs screenshots/before-forside.png og screenshots/before-detail.png. Beskriv konkret hvad du ser. Stemmer det med specifikationen fra fase 3? Er der noget der ser brudt, forkert eller "AI-genereret klodset" ud?
Claude åbner billederne, beskriver layout, indhold og evt. problemer den selv kan se. Den agentiske loop lukker mellem "jeg skrev kode" og "jeg ved hvordan det ser ud".
Pointen: Du har nu en test der gør samme arbejde som auto-verify gør — men eksplicit, gemt i koden, og klar til at køre igen. Det er den slags artefakt der gør forskellen mellem "Claude byggede noget" og "vi har et system".
5

Solo: frontend-design skill + før/efter-sammenligning

Du har en app der virker og en Playwright-test der beviser det. Nu skal du løfte designet — og lade Claude dokumentere ændringen visuelt.

Solo-opgave

Redesign UI'en og lav en før/efter-side

Du installerer frontend-design-skillen, beder Claude redesigne UI'en, kører Playwright igen for "efter"-screenshots, og samler det hele i en HTML-side der viser forskellen side om side.

5.1 — Installer frontend-design skillen
Klik + ved siden af prompt-feltet → Plugins → Add plugin. Søg efter frontend-design i marketplacen og installer.
Alternativt: bed Claude tilføje skillen via prompt — så installerer den den i .claude/skills/ i projektet.
Når skillen er aktiv kan du se den under Customize → Skills i sidebaren.

Skillen er Anthropics egen samling af principper for "ikke at lave generisk AI-look" — typografi, spacing, hierarki, farvevalg.

5.2 — Redesign med skillen aktiv
Redesign frontend'en til "Min medlemsportefølje" med frontend-design-skillen aktiv. Behold funktionaliteten, men løft kvaliteten af typografi, layout og hierarki. Brug en seriøs, dæmpet visuel tone — det er et internt værktøj, ikke en forbrugerapp.
Claude opdaterer CSS/komponenter, kører dev-serveren igen, og du ser en mærkbart mere moden UI: konsistent spacing, ordentlig type-skala, gennemtænkte farver.
5.3 — Kør Playwright igen og lav en før/efter-side
Kør Playwright-testen igen, men gem screenshots som screenshots/after-forside.png og screenshots/after-detail.png. Bagefter: lav en standalone HTML-fil "compare.html" i projektets rod der viser before- og after-billederne side om side i et to-kolonne layout (forside vs forside, detail vs detail), med tydelige overskrifter. Åbn compare.html i Preview-panelet bagefter.
Du får fire screenshots i alt, en compare.html i projektet, og når du åbner den ser du forskellen mellem default-UI og den frontend-design-skill-genererede UI side om side.
Læringen: Skills hæver kvaliteten gratis, og Playwright giver dig et reproducérbart visuelt spor. Sammen har du noget du kan vise til en kollega — eller bruge til at argumentere for at samme skill skal være standard i jeres team.
5.4 — Stretch: Tilføj en ekstern datakilde

Hvis du har tid og lyst — udvid appen med data fra en ekstern API. Forslag:

  • Danmarks Statistik StatBank API — fx pensionsformue eller arbejdsmarkedstal som benchmark
  • Nationalbankens statistik-API — IP-statistik eller renter
Find en relevant ekstern API (fx Danmarks Statistik) og tilføj et benchmark-tal til medlems-detailsiden, så jeg kan sammenligne et medlems opsparing mod et landsgennemsnit. Hvis API'et kræver en nøgle, så læg den i en .env-fil og sørg for at den ikke committes.
Når Claude rører ved .env, hemmeligheder eller eksterne kald, så sig stop og tænk en gang: hvad kan en kombination af "læser data + ekstern netværksadgang + utrusted input" føre til? Den tankerække skal vi tale om bagefter — du skal bare have plantet kimen nu.
